# 索引数据逻辑说明

## 您的理解总结

您的理解是**正确的**，但需要一些补充说明：

### ✅ 正确的部分

1. **每个飞机构型一个标准数据库** ✅
   - 每个构型（Configuration）都有自己的 `index_data` 数据
   - 通过 `configuration_id` 关联
   - 结构相同但数据独立

2. **扁平化的数据表** ✅
   - `index_data` 表以扁平化方式存储层级关系
   - 每条记录是一个完整的"主区域-主部件-一级子部件-二级子部件"路径

3. **数据字段相互关联** ✅
   - 4个层级字段：主区域 → 主部件 → 一级子部件 → 二级子部件
   - 5个独立对照字段：方位、缺陷主体、缺陷描述、位置、数量

4. **用于数据清洗** ✅
   - 索引数据作为"标准对照表"
   - 用于清洗历史工卡数据和缺陷清单数据

### 📝 补充说明

#### 数据清洗流程

根据需求文档 `information.md` 的 FR-DATA-03：

```
原始工卡数据
    ↓
1. 字段标准化（依据索引数据）
   - 匹配"主区域-主部件-一级子部件-二级子部件"层级
   - 标准化字段命名和格式
    ↓
2. AI对比校验（LLM辅助）
   - 使用索引数据作为"标准答案"
   - 检查描述字段的一致性
   - 标记异常数据
    ↓
3. 分类存储
   - 按构型分类
   - 存储到工卡基础数据库
```

#### 相似度计算

索引数据还用于缺陷清单和工卡的相似度计算（见 `similarity_service.py`）：

```python
# 提取层级信息
主区域 → 权重 0.10
主部件 → 权重 0.15
一级子部件 → 权重 0.20
二级子部件 → 权重 0.35  # 最重要
方位 → 权重 0.05
缺陷主体 → 权重 0.10
缺陷描述 → 权重 0.05  # 文本相似度
```

## 数据关系图

```
Configuration (构型)
├─ id, name, aircraft_type, version
├─ field_mapping (独立对照字段: orientation, defectSubject, ...)
│
└─ index_data (层级索引数据)
   ├─ main_area (主区域)
   ├─ main_component (主部件)
   ├─ first_level_subcomponent (一级子部件)
   ├─ second_level_subcomponent (二级子部件)
   ├─ orientation (方位)
   ├─ defect_subject (缺陷主体)
   ├─ defect_description (缺陷描述)
   ├─ location (位置)
   └─ quantity (数量)
```

## 核心价值

索引数据的三个核心价值：

1. **标准化** 📊
   - 统一的字段命名和层级结构
   - 确保不同来源数据的一致性

2. **清洗基准** 🧹
   - 为AI对比校验提供标准答案
   - 识别和修正异常数据

3. **匹配增强** 🎯
   - 提升相似度计算的准确性
   - 通过权重配置优化匹配结果
