# 本地清洗与匹配工作逻辑说明

## 一、整体架构

本地清洗与匹配是一个**独立的工作链路**，与AI链路完全分离：

- ✅ **独立数据表**：使用 `workcard_clean_local`、`defect_clean_local`、`defect_match_local` 等独立表
- ✅ **独立API**：所有接口命名包含 `local`，避免误调用
- ✅ **独立流程**：清洗和匹配逻辑完全独立，不依赖AI链路
- ✅ **最终回归**：匹配完成后回归到现有导入链路（写入 `import_batches`）

## 二、本地清洗逻辑

### 2.1 清洗引擎核心方法

**方法名**：`clean_jobcard_en(text_en: str, compiled: CompiledDict)`

**输入**：
- `text_en`：英文描述文本
- `compiled`：编译后的关键词字典（按构型区分）

**输出**：结构化5维数据
- `main_component`：主部件（多值，逗号分隔）
- `sub_component`：子部件（多值，逗号分隔）
- `location`：位置（单值）
- `orientation`：方位（单值）
- `status`：状态（单值）
- `action`：动作（单值）

### 2.2 清洗规则

#### 规则1：仅英文参与
- ✅ 中文不参与抽取
- ✅ 只处理英文描述文本

#### 规则2：主部件（main_component）
- **多值匹配**：所有匹配的关键词都用逗号分隔存储
- **匹配方式**：使用单词边界匹配（`\b`），确保完整单词匹配
- **示例**：`"SEAT, BIN"` 表示匹配到两个主部件关键词

#### 规则3：子部件（sub_component）
- **多值匹配**：所有匹配的关键词都用逗号分隔存储
- **语境绑定**：在主部件语境下识别子部件
- **智能去重**：识别组合词重复（如 "SEAT BELT" 和 "BELT"）
- **示例**：`"BELT, BUCKLE"` 表示匹配到两个子部件关键词

#### 规则4：位置（location）
- **单值**：取最长匹配的关键词
- **语境绑定**：在主部件语境下识别位置
- **示例**：`"CABIN"` 或 `"COCKPIT"`

#### 规则5：方位（orientation）
- **单值**：取最长匹配的关键词
- **语境绑定**：在主部件语境下识别方位
- **示例**：`"LEFT"` 或 `"RIGHT"`

#### 规则6：状态（status）
- **单值**：全局词表（不绑定主部件）
- **匹配规则**：命中多个时取最长关键词
- **示例**：`"BROKEN"`、`"DIRTY"`、`"FAILED"`

#### 规则7：动作（action）
- **单值**：全局词表（不绑定主部件）
- **匹配规则**：命中多个时取最长关键词
- **示例**：`"REPLACE"`、`"REPAIR"`、`"CLEAN"`

### 2.3 清洗流程

```
输入：英文描述文本 + 关键词字典
  ↓
1. 提取主部件（main_component）- 多值匹配
  ↓
2. 对每个主部件，提取：
   - 子部件（sub_component）- 多值匹配
   - 位置（location）- 单值，最长匹配
   - 方位（orientation）- 单值，最长匹配
  ↓
3. 提取全局字段：
   - 状态（status）- 单值，最长匹配
   - 动作（action）- 单值，最长匹配
  ↓
4. 去重和格式化
  ↓
输出：结构化5维数据
```

## 三、智能匹配逻辑

### 3.1 匹配流程概述

```
输入：缺陷清洗结果 + 工卡清洗结果
  ↓
1. 按主部件建立索引（加速查找）
  ↓
2. 对每条缺陷，查找相同主部件的工卡
  ↓
3. 对每对缺陷-工卡进行维度比对打分
  ↓
4. 计算智能关键词匹配奖励
  ↓
5. 按总分降序排序
  ↓
6. 保存结果（≥85分保存到数据库，≥50分返回前端）
```

### 3.2 评分权重体系

总分：**100分**

| 维度 | 权重 | 说明 |
|------|------|------|
| main_component | 32.55分 | 主部件匹配（原35分×0.93） |
| sub_component | 46.5分 | 子部件匹配（原50分×0.93） |
| location | 2.79分 | 位置匹配（原3分×0.93） |
| orientation | 1.86分 | 方位匹配（原2分×0.93） |
| status | 4.65分 | 状态匹配（原5分×0.93） |
| action | 4.65分 | 动作匹配（原5分×0.93） |
| **keyword_match_bonus** | **7.0分** | **智能匹配奖励（新增）** |

### 3.3 主部件匹配策略

#### 情况1：双方都没有主部件
- **结果**：返回 `None`（不匹配）

#### 情况2：只有一方有主部件
- **得分**：20.0分（部分匹配）

#### 情况3：双方都有主部件
- **计算交集**：比较双方主部件关键词集合
- **完全匹配**（所有关键词都匹配）：32.55分（满分）
- **匹配2个及以上关键词**：30.0分（高分）
- **匹配1个关键词**：25.0分（中等分）
- **无交集**：返回 `None`（不匹配）

### 3.4 子部件匹配策略（混合策略）

#### 情况1：双方都没有子部件
- **特殊处理**：将子部件分数上限（46.5分）分配给其他有匹配的字段
- **分配规则**：平均分配给所有有匹配的字段（main、location、orientation、status、action、keyword_match）
- **示例**：如果有3个字段匹配，每个字段额外获得 46.5÷3 = 15.5分

#### 情况2：双方都有子部件
- **匹配2个及以上关键词**：46.5分（满分）
- **匹配1个关键词**：41.85分
- **无交集**：返回 `None`（不匹配）

#### 情况3：只有一方有子部件
- **得分**：25.0分（部分匹配）

### 3.5 智能关键词匹配奖励（核心创新）

**算法名称**：智能关键词匹配奖励算法 2.0

**总分**：7分

**计算方式**：

#### 步骤1：文本分词
- 使用正则表达式 `\b\w+\b` 提取所有单词
- 保留单字母单词（如 P、L、R）
- 转换为小写并去重

#### 步骤2：计算覆盖率（Recall）- 5分池
- **覆盖率** = 交集单词数 / 目标文本单词数
- **≥80%覆盖率**：5.0分
- **≥70%覆盖率**：3.0分
- **<70%覆盖率**：0分

#### 步骤3：计算紧凑度（Precision）- 2分池
- **紧凑度** = 交集单词数 / 候选文本单词数
- **得分** = 紧凑度 × 2.0（最高2.0分）

#### 步骤4：归一化
- **总奖励分** = 覆盖率得分 + 紧凑度得分
- **归一化值** = 总奖励分 / 7.0（返回0-1范围）
- **最终得分** = 归一化值 × 7.0

**示例**：
```
目标文本："SEAT BELT BROKEN"
候选文本："REPLACE SEAT BELT BUCKLE"

分词：
- 目标：{seat, belt, broken}
- 候选：{replace, seat, belt, buckle}
- 交集：{seat, belt}

覆盖率 = 2/3 = 66.7% → 0分（<70%）
紧凑度 = 2/4 = 50% → 1.0分（50% × 2）
总奖励 = 0 + 1.0 = 1.0分
最终得分 = 1.0分
```

### 3.6 其他维度匹配

- **location**：完全相同则得2.79分，否则0分
- **orientation**：完全相同则得1.86分，否则0分
- **status**：完全相同则得4.65分，否则0分
- **action**：完全相同则得4.65分，否则0分

### 3.7 匹配阈值

| 阈值 | 用途 | 说明 |
|------|------|------|
| **≥85分** | 保存到数据库 | 只有高分候选才持久化存储 |
| **≥50分** | 前端展示 | 所有候选都返回给前端展示 |
| **<50分** | 不匹配 | 返回 `None`，不参与后续处理 |

### 3.8 性能优化

#### 优化1：主部件索引
- 在匹配前，按主部件建立索引字典
- 只对相同主部件的缺陷-工卡进行匹配
- **效果**：大幅减少匹配次数，提升性能

#### 优化2：批量处理
- 一次性处理所有缺陷记录
- 批量保存匹配结果
- **效果**：减少数据库交互次数

## 四、工作流程

### 4.1 完整流程

```
1. 关键词字典管理
   ↓
2. 工卡清洗（历史工卡或上传Excel）
   ↓
3. 缺陷清洗（缺陷记录）
   ↓
4. 智能匹配
   ├─ 按主部件索引
   ├─ 维度比对打分
   ├─ 智能关键词匹配奖励
   └─ 结果排序
   ↓
5. 结果保存
   ├─ ≥85分：保存到数据库
   └─ ≥50分：返回前端展示
```

### 4.2 数据流转

```
原始数据（英文描述）
  ↓ [本地清洗]
结构化数据（5维）
  ↓ [智能匹配]
匹配结果（分数+候选工卡）
  ↓ [用户确认]
导入批次（import_batches）
```

## 五、关键特性

### 5.1 多值匹配
- **主部件**和**子部件**支持多值匹配
- 提高召回率，避免遗漏匹配

### 5.2 智能匹配奖励
- 基于文本相似度的7分奖励机制
- 提升匹配准确性，减少误匹配

### 5.3 子部件分数分配
- 双方都没有子部件时，将分数分配给其他匹配字段
- 避免因缺少子部件而丢失匹配机会

### 5.4 阈值分级
- 数据库保存阈值（85分）与前端展示阈值（50分）分离
- 平衡存储成本和用户体验

### 5.5 语境绑定
- 子部件、位置、方位在主部件语境下识别
- 提高识别准确性

## 六、技术实现细节

### 6.1 关键词匹配方法

**单词边界匹配**：
```python
pattern = r'\b' + re.escape(keyword) + r'\b'
```
- 使用 `\b` 确保只匹配完整单词
- 例如：`'BIN'` 能匹配 `'THE BIN IS BROKEN'` 但不能匹配 `'CABIN'`

### 6.2 去重算法

**子部件去重**：
- 识别组合词重复（如 "SEAT BELT" 和 "BELT"）
- 优先保留单独词（关键词匹配优先）
- 避免重复存储

### 6.3 分数计算精度

- 所有分数计算保留2位小数
- 使用 `round()` 函数确保精度

### 6.4 数据库存储

**保存字段**：
- `score_total`：总分
- `score_main`：主部件得分
- `score_sub`：子部件得分
- `score_location`：位置得分
- `score_orientation`：方位得分
- `score_status`：状态得分
- `score_action`：动作得分
- **注意**：`score_keyword_match` 不保存到数据库（模型中没有该字段）
