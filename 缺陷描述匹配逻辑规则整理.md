# 缺陷描述匹配逻辑规则整理

## 一、概述

本系统包含两套匹配逻辑：
1. **LLM匹配逻辑**：基于大语言模型的智能匹配（用于AI匹配）
2. **本地匹配逻辑**：基于关键词字典的本地清洗匹配（用于本地匹配）

---

## 二、LLM匹配逻辑（AI匹配）

### 2.1 匹配流程

**文件位置**：`backend/app/api/api_v1/endpoints/matching.py`

**匹配方式**：使用大语言模型（LLM）进行语义理解和匹配

### 2.2 字段权重体系

| 字段 | 权重 | 说明 |
|------|------|------|
| 主区域 | 10% | 基础匹配字段 |
| 主部件 | 15% | 重要匹配字段 |
| 一级子部件 | 20% | 重要匹配字段 |
| **二级子部件** | **35%** | **最重要字段** |
| 方位 | 5% | 辅助匹配字段 |
| 缺陷主体 | 10% | 辅助匹配字段 |
| 缺陷描述 | 5% | 辅助匹配字段 |

### 2.3 匹配优先级规则

#### 优先级1：主区域+主部件+二级子部件都匹配
- **相似度范围**：80-100分
- **说明**：最高优先级，完全匹配

#### 优先级2：主区域+主部件匹配
- **相似度范围**：60-79分
- **说明**：重要字段匹配，但缺少二级子部件

#### 优先级3：主区域匹配
- **相似度范围**：40-59分
- **说明**：仅基础字段匹配

### 2.4 重要约束规则

1. **主区域必须匹配**
   - 如果主区域不匹配，相似度必须为0分
   - 不能返回该工卡

2. **主部件约束**
   - 如果主区域匹配但主部件不匹配，相似度不能超过60分

3. **层级递进关系**
   - 必须严格遵循层级递进关系
   - 不能跨层级匹配

### 2.5 语义匹配规则

#### 支持同义词识别
- 驾驶舱 = Cockpit = 驾驶室 = 驾驶舱区域
- 客舱 = Cabin = 客舱区域
- 座椅 = Seat = 座位
- 安全带 = Safety Belt = Seat Belt = 安全扣带

#### 主区域匹配严格性
- 主区域必须完全匹配或同义词匹配
- **"客舱"不能匹配"驾驶舱"**
- **"驾驶舱"不能匹配"客舱"**
- 不同主区域之间不能进行语义匹配

#### 主部件和子部件匹配
- 支持部分匹配：如果描述中包含索引数据中的关键词，可以进行部分匹配
- **但必须在同一主区域下进行匹配**

#### 严格约束
- 不能跨主区域匹配（例如：客舱的部件不能匹配驾驶舱的工卡）
- 语义理解不能违反层级递进关系

### 2.6 匹配步骤

1. 首先检查主区域是否匹配（支持同义词）
2. 然后检查主部件是否匹配（支持同义词）
3. 再检查一级子部件是否匹配（支持同义词）
4. 重点检查二级子部件是否匹配（权重最高）
5. 最后检查其他字段（方位、缺陷主体、缺陷描述等）
6. 根据匹配的字段和权重计算相似度分数

## 三、本地匹配逻辑（本地清洗匹配）

### 3.1 匹配流程

**文件位置**：`backend/app/services/localwash_service.py`

**匹配方式**：基于关键词字典的维度比对打分

### 3.2 评分权重体系

总分：**100分**

| 维度 | 权重 | 说明 |
|------|------|------|
| main_component | 32.55分 | 主部件匹配（原35分×0.93） |
| sub_component | 46.5分 | 子部件匹配（原50分×0.93） |
| location | 2.79分 | 位置匹配（原3分×0.93） |
| orientation | 1.86分 | 方位匹配（原2分×0.93） |
| status | 4.65分 | 状态匹配（原5分×0.93） |
| action | 4.65分 | 动作匹配（原5分×0.93） |
| **keyword_match_bonus** | **7.0分** | **智能匹配奖励（新增）** |

### 3.3 主部件匹配策略

#### 情况1：双方都没有主部件
- **结果**：返回 `None`（不匹配）

#### 情况2：只有一方有主部件
- **得分**：20.0分（部分匹配）

#### 情况3：双方都有主部件
- **计算交集**：比较双方主部件关键词集合（多值，逗号分隔）
- **完全匹配**（所有关键词都匹配）：32.55分（满分）
- **匹配2个及以上关键词**：30.0分（高分）
- **匹配1个关键词**：25.0分（中等分）
- **无交集**：返回 `None`（不匹配）

**示例**：
```
缺陷主部件："SEAT, BIN"
工卡主部件："SEAT, BIN, TABLE"
交集：{"SEAT", "BIN"}
匹配结果：30.0分（匹配2个关键词）
```

### 3.4 子部件匹配策略（混合策略）

#### 情况1：双方都没有子部件
- **特殊处理**：将子部件分数上限（46.5分）分配给其他有匹配的字段
- **分配规则**：平均分配给所有有匹配的字段（main、location、orientation、status、action、keyword_match）
- **示例**：如果有3个字段匹配，每个字段额外获得 46.5÷3 = 15.5分

#### 情况2：双方都有子部件
- **匹配2个及以上关键词**：46.5分（满分）
- **匹配1个关键词**：41.85分
- **无交集**：返回 `None`（不匹配）

#### 情况3：只有一方有子部件
- **得分**：25.0分（部分匹配）

### 3.5 智能关键词匹配奖励算法

**算法名称**：智能关键词匹配奖励算法 2.0

**总分**：7分

#### 计算步骤

**步骤1：文本分词**
- 使用正则表达式 `\b\w+\b` 提取所有单词
- 保留单字母单词（如 P、L、R）
- 转换为小写并去重

**步骤2：计算覆盖率（Recall）- 5分池**
- **覆盖率** = 交集单词数 / 目标文本单词数
- **≥80%覆盖率**：5.0分
- **≥70%覆盖率**：3.0分
- **<70%覆盖率**：0分

**步骤3：计算紧凑度（Precision）- 2分池**
- **紧凑度** = 交集单词数 / 候选文本单词数
- **得分** = 紧凑度 × 2.0（最高2.0分）

**步骤4：归一化**
- **总奖励分** = 覆盖率得分 + 紧凑度得分
- **归一化值** = 总奖励分 / 7.0（返回0-1范围）
- **最终得分** = 归一化值 × 7.0

#### 计算示例

```
目标文本："SEAT BELT BROKEN"
候选文本："REPLACE SEAT BELT BUCKLE"

分词：
- 目标：{seat, belt, broken}
- 候选：{replace, seat, belt, buckle}
- 交集：{seat, belt}

覆盖率 = 2/3 = 66.7% → 0分（<70%）
紧凑度 = 2/4 = 50% → 1.0分（50% × 2）
总奖励 = 0 + 1.0 = 1.0分
最终得分 = 1.0分
```

### 3.6 其他维度匹配

| 维度 | 匹配规则 | 得分 |
|------|----------|------|
| location | 完全相同 | 2.79分 |
| orientation | 完全相同 | 1.86分 |
| status | 完全相同 | 4.65分 |
| action | 完全相同 | 4.65分 |

**注意**：所有维度都是精确匹配，不完全相同则得0分。

### 3.7 匹配阈值

| 阈值 | 用途 | 说明 |
|------|------|------|
| **≥85分** | 保存到数据库 | 只有高分候选才持久化存储 |
| **≥50分** | 前端展示 | 所有候选都返回给前端展示 |
| **<50分** | 不匹配 | 返回 `None`，不参与后续处理 |

### 3.8 性能优化

#### 优化1：主部件索引
- 在匹配前，按主部件建立索引字典
- 只对相同主部件的缺陷-工卡进行匹配
- **效果**：大幅减少匹配次数，提升性能

#### 优化2：批量处理
- 一次性处理所有缺陷记录
- 批量保存匹配结果
- **效果**：减少数据库交互次数

## 四、相似度服务（SimilarityService）

### 4.1 服务位置

**文件位置**：`backend/app/services/similarity_service.py`

### 4.2 计算方式

#### 方式1：基于索引数据的相似度计算（优先）

**权重体系**：
- 主区域：4%
- 主部件：19%
- 一级子部件：34%
- 二级子部件：19%
- 方位：4%
- 缺陷主体：9%
- 缺陷描述：4%
- 智能关键词匹配奖励：7%

**字段相似度计算**：
- 完全匹配：1.0
- 包含匹配：0.8
- 词汇相似度：Jaccard相似度（交集/并集）

#### 方式2：基础字段匹配（向后兼容）

**权重体系**：
- system：30%
- component：30%
- description：40%

### 4.3 智能关键词匹配奖励

**算法**：与本地匹配逻辑相同（智能关键词匹配奖励算法 2.0）

**计算方式**：
1. 文本分词（使用 `\b\w+\b` 正则）
2. 计算覆盖率（Recall）- 5分池
3. 计算紧凑度（Precision）- 2分池
4. 归一化到0-1范围

## 五、匹配逻辑对比

### 5.1 LLM匹配 vs 本地匹配

| 特性 | LLM匹配 | 本地匹配 |
|------|---------|----------|
| **匹配方式** | 语义理解 | 关键词比对 |
| **权重体系** | 7个字段 | 7个维度 |
| **最高权重** | 二级子部件（35%） | 子部件（46.5分） |
| **智能奖励** | 无 | 7分智能匹配奖励 |
| **阈值** | 按优先级划分 | ≥85分保存，≥50分展示 |
| **同义词支持** | 是 | 否（精确匹配） |
| **跨区域匹配** | 禁止 | 禁止 |

### 5.2 适用场景

**LLM匹配**：
- 需要语义理解的场景
- 支持同义词识别
- 适合复杂语义匹配

**本地匹配**：
- 需要快速匹配的场景
- 基于关键词字典
- 适合批量处理

## 六、关键约束总结

### 6.1 通用约束

1. **主区域必须匹配**
   - 主区域不匹配，相似度为0分
   - 不能跨主区域匹配

2. **层级递进关系**
   - 必须遵循层级递进关系
   - 不能跨层级匹配

3. **精确匹配要求**
   - 本地匹配要求精确匹配（完全相同）
   - LLM匹配支持同义词识别

### 6.2 特殊规则

1. **子部件分数分配**（仅本地匹配）
   - 双方都没有子部件时，将46.5分分配给其他匹配字段

2. **智能匹配奖励**（本地匹配和相似度服务）
   - 基于文本相似度的7分奖励机制
   - 提升匹配准确性

3. **多值匹配**（仅本地匹配）
   - 主部件和子部件支持多值匹配
   - 使用逗号分隔存储
