飞机方案处理系统 需求说明文档

1. 引言

1.1 项目目标

本项目旨在构建一个智能化的飞机方案处理系统，通过引入大模型（LLM）技术对历史工卡数据进行高效清洗、分类，并实现用户提交的飞机缺陷清单与历史工卡数据的智能对比和匹配，最终辅助用户完成工卡录入流程。

1.2 范围定义

本系统主要覆盖数据准备（历史数据清洗）和业务处理（缺陷清单对比、候选工卡推荐及录入交互）两大核心功能。不包括公司工卡系统本身的后台修改和运行逻辑，仅负责数据交互录入。

3. 功能需求 (FR)

3.1 模块一：历史工卡数据清洗与存储 (FR-DATA)

该模块负责将原始的、未分类的历史工卡数据转化为结构化、高质量的、按构型分类的工卡基础数据库。

FR-DATA-01：构型索引文件加载与选择

系统应提供界面，允许管理员加载和选择不同构型对应的索引文件（Index File），以确定清洗目标数据的结构。

索引文件在界面上通过树形编辑器的模式做编辑。并能保存模板和另存为新的数据表。因为不同的飞机构型，对应着不同的数据索引表文件

FR-DATA-02：工卡数据分类选择

系统应根据选定的构型索引文件，从历史数据库中筛选出符合该构型结构的飞机工卡数据表。

FR-DATA-03：数据分类清洗（LLM辅助）

系统应根据选定的工卡数据，执行分类清洗流程：

数据字段标准化： 依据索引文件，将原始工卡数据字段映射至标准字段。

AI对比校验： 引入大模型AI对工卡数据的关键描述字段（例如，故障描述、执行动作等）进行内容对比和语义校验，确保同构型下的数据一致性高，并标记潜在的异常数据点。大模型用demo2文件夹中的qwen模型，代码和apikey是已有可用的保存构型失败: Failed to fetch

结果分类： 清洗后的数据应按构型进行严格分类。

FR-DATA-04：清洗结果存储

清洗完成后的结构化工卡数据，应存储至工卡基础数据库中，作为后续缺陷对比的基准数据源。

3.2 模块二：缺陷清单处理与工卡匹配 (FR-DEFECT)

该模块负责处理用户提交的缺陷清单，进行匹配计算，并提供人机交互界面进行最终确认和录入。

FR-DEFECT-01：缺陷清单提交与存储

用户应能通过界面提交缺陷清单数据。

提交后，系统应首先根据飞机所属构型的索引文件，对清单数据进行分类清洗（可复用FR-DATA-03的AI对比校验逻辑）。

清洗后的缺陷清单数据应按飞机号（例如，B-XXXX）单独存入飞机缺陷清单数据库。

FR-DEFECT-02：核心工卡对比算法

系统需实现一个字段递进式对比算法，用于评估缺陷清单中的每一条记录与工卡基础数据库（Module 1的产出）中对应构型工卡记录的相似度。

筛选： 对比前，系统应根据缺陷清单记录的关键信息（如系统、部件、位置等）对工卡基础数据库做第一级筛选。

递进对比： 对比过程应分字段进行，字段间存在递进关系（例如，系统 -> 部件 -> 故障描述）。

权重计算： 每一个对比层级（字段）都应设置权重参数（可配置），相似度评分应是基于这些权重参数的加权求和。

相似度评分： 最终输出一个相似度评分（0-100分）。

FR-DEFECT-03：候选工卡清单推荐

系统应将与缺陷清单中任一条记录相似度评分高于 75 分的项目，列举到候选工卡清单中。

FR-DEFECT-04：用户选择与工卡录入交互

人机交互： 界面应展示缺陷清单中的单条记录及其对应的候选工卡清单。

单项选择： 用户必须对每个缺陷记录从候选清单中单项选择一个最匹配的工卡。

后台录入： 选定后，系统应触发后台程序：

将缺陷清单的内容和所选工卡清单的内容进行数据整合。

通过 API 或数据接口向公司的工卡系统执行交互录入操作。

FR-DEFECT-05：落选清单管理与导出

落选展示： 界面应清晰显示落选的缺陷清单内容（即所有未被选中或相似度评分低于75分的缺陷记录）。

导出功能： 系统需提供导出落选缺陷清单的功能（例如，导出为CSV或Excel格式），方便用户进行人工复核或二次处理。

4. 非功能需求 (NFR)

4.1 性能要求

数据清洗速度： 历史数据清洗模块（FR-DATA）应具备高吞吐量，能在一小时内处理至少1GB的历史工卡数据。

匹配响应时间： 缺陷清单对比（FR-DEFECT-02）的核心匹配算法，应在用户提交清单后的 5 秒内返回候选工卡清单。

4.2 可用性与界面 (UX)

界面应采用现代化、响应式设计，确保在不同设备上（桌面、平板）都能流畅操作。

核心业务流程（缺陷清单处理）的界面应逻辑清晰，支持分步操作，并提供进度反馈。

4.3 安全性

所有数据库连接和数据交互（尤其是与公司工卡系统的后台交互）必须进行严格的身份验证和授权。

数据存储和传输应符合公司的数据安全和隐私政策。

5. 技术要求与架构考虑

5.1 数据存储

工卡基础数据库： 用于存储清洗后的历史工卡数据，要求具备高并发读写能力和复杂查询支持（推荐使用高性能关系型数据库或文档型数据库）。

飞机缺陷清单数据库： 独立存储用户提交的、按飞机号分类的缺陷数据。

5.2 大模型集成 (LLM)

LLM服务应以 API 形式接入，负责语义理解、数据校验和清洗过程中的内容对比。

LLM仅作为清洗和校验工具，不参与核心的相似度评分算法（FR-DEFECT-02），核心算法应由确定性的加权逻辑实现。

5.3 后台交互

与公司工卡系统的交互（FR-DEFECT-04）应通过可靠的 API 或消息队列进行，确保数据录入的原子性和可追溯性。

5.4 算法配置

相似度评分算法中的字段权重参数（FR-DEFECT-02）必须是可配置的，以便运营人员根据业务反馈调整匹配准确性。
